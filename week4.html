Week 4: Target Systems (Part 1)

Explore architectural design mechanics, structural patterns, and adaptive lifecycle strategies within the Object-Oriented Paradigm.

I. Task Definition: The Foundation

Task definition is the initial phase where software requirements are precisely defined. It acts as the "North Star" for the entire development process.

The Process

  1. Requirement Gathering: Collecting detailed information from stakeholders (what must the software do?).
  2. Specification Creation: Converting those needs into a detailed document outlining functionality and constraints.

Role in OOAD

  • Goal Alignment: Ensures that modelled objects and interactions actually solve the user's business goals.
  • Relation to OOA:
    • Information Domain: It clarifies which entities (like players or bank accounts) are relevant to the problem.
    • Describing Function: It defines the "To-Do" list that determines what methods a class needs.

Extended Pedagogical Commentary

From an architectural point of view, skipping formal Task Definition causes scope creep. In Object-Oriented Analysis (OOA), raw requirements are translated into classes and relationships. Without accurate software requirement collection, system engineers risk writing bloated APIs that do not serve real business scenarios.

II. Top-Down vs. Bottom-Up Approaches

These are general strategies for tackling complexity. They are not exclusive to OOP but are essential tools for structuring a design.

1. Top-Down Approach (Decomposition)

  • Process: Starts with the highest level of the system (the "Big Picture") and progressively breaks it down into smaller, detailed sub-parts.
  • OOD Role: Helps define the overall system architecture and major classes first.
  • OOA Link: Relates to Dividing Models for Detail. You start with the most general system and refine it.

2. Bottom-Up Approach (Integration)

  • Process: Starts with the most fundamental, specific parts (basic classes) and combines them to form higher-level components.
  • OOD Role: Useful when specifications aren't fully defined; allows for testing small parts immediately.
  • OOA Link: Relates to Starting Simple, Getting Detailed. You build and test the core "behaviours" before worrying about the massive system framework.

Systemic Contrast

Top-Down decomposition relies on abstract structural patterns and system layout maps, while Bottom-Up software strategies are highly iterative, relying on the unit testing of modular concrete classes before assembling them into a broader system application layer.

III. The Facade Pattern (A Structural Design Pattern)

This is a critical Object-Oriented Design (OOD) feature used to manage complexity.

Definition

A Facade is a "wrapper" class that provides a simplified interface to a large, complex body of code (a subsystem).

Purpose

  • Ease of Use: It provides a high-level interface so external code doesn't have to deal with dozens of complex internal classes.
  • Reduced Dependency: External modules only need to know how to talk to the Facade, not the internal "guts" of the subsystem.

OOAD Application

  • Representing Behaviour: It hides complex internal behaviours behind a simple "button" or method call.
  • Dividing Models: It helps separate complex sections, keeping internal complexities hidden from outer modules.

IV. Agility: The Adaptive Philosophy

Agility is the ability to respond to change and unpredictability. It is a philosophy, not a technical feature of OOD.

Characteristics

  • Iterative Development: Developing software in small, frequent increments rather than one massive release.
  • Feedback Loops: Revisiting and revising decisions based on user feedback.

Relation to OOAD

  • OOD Link: Supports the iterative refinement of object systems.
  • OOA Link (Starting Simple, Getting Detailed): Agile development perfectly matches the OOA principle of starting with a simple model and adding complexity only when needed.

V. Technical Revision Table (Exam Preparation)

Methodology Definition/Key Focus OOA/OOD Relation
Task Definition Precise requirements gathering. Defines the Information Domain.
Top-Down High-level → Detailed. Useful for Dividing Models.
Bottom-Up Detailed parts → Integrated System. Useful for Starting Simple.
Facade Pattern Simplified interface for complex guts. A structural feature of OOD.

VI. Synergy: How they work together

In a real project, these are not isolated. A developer might:

  1. Use Agility to plan short bursts of work.
  2. Use Task Definition to decide what that work is.
  3. Use a Top-Down approach to sketch the classes.
  4. Use a Facade to make sure that new, complex code doesn't break the rest of the system.

Exam Theory Tip: If asked to differentiate between Top-Down and Bottom-Up, remember: Top-Down is about breaking the system apart to understand it, while Bottom-Up is about building pieces together to create it.

VII. Analytical Exam Assessment Suite

Review these highly targeted short answer questions to verify your technical understanding of the theoretical systems concepts taught in Week 4.

Question 1: Architectural Comparison 4 Marks

Differentiate between Top-Down decomposition and Bottom-Up integration during the software engineering design phase. Explain how each approach uniquely affects Object-Oriented Analysis (OOA).

Detailed Sample Solution:

Top-Down decomposition and Bottom-Up integration are distinct architectural pathways for managing system complexity:

  • Top-Down Approach: Starts with the highest macro level of the software system (the "Big Picture") and systematically decomposes it into small, granular modular components. In OOA, this directly aligns with Dividing Models for Detail, where engineers conceptualise the foundational system framework first before mapping individual subclasses.
  • Bottom-Up Approach: Commences at the micro level, implementing fundamental, highly specific components (individual low-level classes) and integrating them upwards into complex high-level structures. In OOA, this supports Starting Simple, Getting Detailed, allowing developers to implement and test core object behaviours immediately without waiting for global architectural parameters to be finalised.
Question 2: Design Patterns 3 Marks

Explain how the structural Facade Pattern supports structural encapsulation and simplifies dependency management within an object-oriented application.

Detailed Sample Solution:

The Facade Pattern acts as an architectural wrapper class that exposes a simplified, clean public interface to an underlying complex network of subsystem components. It minimises system coupling in two distinct ways:

  1. Encapsulation of Behaviour: It wraps internal, intricate multi-class execution logic within an isolated uniform method, preventing client applications from interacting directly with internal logic.
  2. Dependency Mitigation: Client models only establish a contract with the unified Facade class rather than maintaining complex references to dozens of discrete inner utility models, reducing global system vulnerabilities.
Question 3: System Lifecycles 3 Marks

Analyse the intersection of Agility and Object-Oriented Analysis (OOA). How does an adaptive philosophy influence class blueprint design cycles?

Detailed Sample Solution:

Agility is an evolutionary development philosophy focusing on iterative design adjustments, continuous user deployment, and rapid adaptation to stakeholder shifts. Within OOA, Agility corresponds directly to the principle of Starting Simple, Getting Detailed. Rather than forcing developers to produce completely finalized class schemas during initial project setup, an agile methodology expects software layouts to mature dynamically. Object models are continuously refactored through brief development increments based on incoming validation data, ensuring functional structures match changing business logic.
