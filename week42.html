Week 4: Target Systems (Part 2)

Advanced Architectural Blueprints: Structural Design Patterns, Domain Modelling, Component Interaction Frameworks, and Python Paradigms.

1. The Facade Pattern: Definition and Purpose

The Facade pattern is a structural design pattern that provides a simplified interface to a complex set of classes, library, or framework (a "subsystem").

  • The Problem: Large systems often have many moving parts (subsystems) that are difficult for a "client" (the user or external code) to use directly.
  • The Solution: A "Facade" class acts as a single point of entry, bundling complex multi-step operations into simple, high-level methods.
  • The Remote-Control Analogy:
    • A Home Theatre has a Projector, Amplifier, and DVD Player (Subsystems).
    • To watch a movie, you'd usually have to turn each on, set resolutions, and adjust volumes manually.
    • The Facade is the "Watch Movie" button on a universal remote. You press one button; it handles the internal complexity of talking to all three devices.

Pedagogical Insight: Managing Systemic Coupling

In modern software engineering, code changes constantly. If your client code interacts directly with twenty individual subsystem classes, any internal modification within those subsystems triggers a catastrophic cascade of errors across your codebase. The Facade pattern establishes an architectural buffer layer, ensuring changes remain isolated behind a unified public interface contract.

2. Key Components of the Facade Pattern

In any exam scenario (School, Library, Theatre), identify these three parts:

Component Role Example (School System)
Client The code that needs a task done. The main() function or Admin UI.
Facade Class The simplified "wrapper" that coordinates tasks. SchoolManagementFacade
Subsystems The complex classes that do the actual work. StudentRecords, TeacherRecords, Schedules.

3. Facade vs. Inheritance

This is a high-probability exam comparison. While both involve hierarchy, their logical "relationships" are completely different.

Inheritance ("Is-A" Relationship)

  • Focus: Extending functionality and code reuse.
  • Structure: A Student is a Person.
  • Purpose: The child class gains the attributes/methods of the parent.

Facade ("Has-A" Relationship)

  • Focus: Simplification and encapsulation.
  • Structure: A SchoolFacade has a StudentRecord subsystem.
  • Purpose: The Facade coordinates the subsystems without inheriting from them. It doesn't "become" the subsystem; it just manages it.

Design Rule of Thumb: Prefer composition (Has-A) over inheritance (Is-A) when creating structural boundaries to prevent fragile base class issues and keep architectural components loosely coupled.

4. Theory of Implementation (How it Works)

You won't be asked to code, but you may be asked to describe the logic flow of a Facade method:

  1. Initialisation: The Facade is created and "holds" instances of all the subsystems it needs to manage.
  2. Coordination: When a method like enrol_student() is called, the Facade:
    • Creates a new Student object.
    • Calls the StudentRecords subsystem to save the data.
    • Might call the ClassSchedules subsystem to assign a default class.
  3. Encapsulation: The Client never sees the StudentRecords or ClassSchedules logic; it only sees the success message from the Facade.

Concrete Python Implementation Model

Observe how the Facade class coordinates disparate subsystems behind a clean public interface layer:

Python Facade Pattern Blueprint Python 3.x 
class StudentRecordsSubsystem:
    def save_to_database(self, student_name):
        print(f"[Subsystem] Saving {student_name} to secure database records...")
        return True

class ClassSchedulesSubsystem:
    def assign_timetable(self, student_name):
        print(f"[Subsystem] Generating default academic timetable for {student_name}...")
        return "Timetable_v1.pdf"

class SchoolManagementFacade:
    def __init__(self):
        # Composition: The Facade "Has-A" reference to subsystems
        self.records = StudentRecordsSubsystem()
        self.schedules = ClassSchedulesSubsystem()

    def enrol_student(self, name):
        # Encapsulating multi-step execution logic
        print(f"\n--- Facade Processing Enrolment for {name} ---")
        success = self.records.save_to_database(name)
        timetable = self.schedules.assign_timetable(name)
        
        if success:
            print("--- Enrolment Operations Complete ---")
            return f"Success! {name} registered with {timetable}."

# Client Code Block
if __name__ == "__main__":
    # Client only interacts with the unified Facade class interface
    school_portal = SchoolManagementFacade()
    status_msg = school_portal.enrol_student("Alice Smith")
    print(status_msg)

5. Benefits of the Facade Pattern (Exam Keywords)

  • Simplified Interface: Users don't need to learn a complex API.
  • Loose Coupling: The client code is not "tied" to the subsystems. If you change how StudentRecords works, you only update the Facade, not every piece of code in the school.
  • Improved Readability: The code becomes more intuitive (e.g., school.hire_teacher() vs. five lines of manual database updates).
  • Layering: It allows for a clean "Message Layer" in the OOD Pyramid.

I. The OOA Game Design Framework

When designing a game, OOAD provides a structured process to ensure the system is scalable (easy to add levels/enemies later).

1. Model the Information Domain (The Entities)

Identify the "nouns" in your game world.

  • Attributes: The static data an object holds (e.g., health, position, inventory, speed).
  • Objects: Player, Enemies, NPCs, and the Environment (obstacles, terrain).

2. Represent Behaviour (The Interactions)

Define how objects respond to internal logic or external triggers.

  • Player Behaviour: Moving based on input, jumping, or interacting with items.
  • Enemy Behaviour: AI patterns like Patrolling (pre-set path), Chasing (moving toward player), or Fleeing.
  • Physics Interaction: What happens during a Collision (stopping) or a Trigger (passing through a "finish line").

3. Describe Function (The Systems)

The high-level "rules" and mechanics that manage the game state.

  • Scoring System: How points are calculated and stored.
  • Health/Lives System: Logic for damage, respawning, and "Game Over" states.
  • Quest Management: Tracking objectives and rewards.

II. Dividing Models for Detail

A crucial theoretical concept is separating your game code into three distinct "Models." This prevents a "spaghetti code" mess where one script does everything.

Model Type Purpose Content Example
Data Models Store the "State" (Information). A class holding int health and float speed.
Functional Models Manage the "Logic" (Mechanics). Scripts for GameManager, ScoreTracker, or InputHandler.
Behavioural Models Define "Reactions" (AI/Physics). AI scripts for an enemy chasing a player.

III. Incremental Development: Simple to Detailed

OOAD logic dictates that you should never build a complex game all at once. Instead, you follow a Development Cycle:

  1. Initial Setup (Prototype): Create basic shapes (cubes/spheres) with basic collision.
  2. Detail Addition: Add "juice"—animations, audio, and basic health.
  3. Expansion: Introduce complex AI (Finite State Machines) and varied player abilities (Power-ups/Weapons).
  4. Refinement: Implement persistent systems like Quest logs and multi-level difficulty scaling.

V. Strategic Design: The "Stealth Game" Scenario

If your exam asks you to design a Stealth Game using OOA:

  • Information Domain: Player (stealth meter, position), Guard (vision cone, patrol points), Environment (hiding spots, noise-making surfaces).
  • Behaviour: Guards patrol until they see the player, then they switch to "Alert" status.
  • Function: Detection system (if stealth meter reaches 0, player is caught), Hiding system (if player is in a "hiding spot," they are invisible to guards).

Theory Tip: Remember that Raycasting is a functional tool used to detect hits (shooting) or sight (AI vision cones). It "shoots" an invisible line and returns data on what it hits. Even if you don't code it, knowing it is a "Function" is key.

I. Raycasting: The Invisible Interaction

Raycasting is the primary "theoretical" tool for shooting mechanics. It is conceptually different from physical projectiles (like a bouncing ball).

  • Definition: Raycasting involves projecting an invisible line (a "ray") from a specific point (the camera) into the 3D scene.
  • The Logic: The system checks if this ray intersects with any Colliders. If it hits something, it returns data about that object (its name, position, and components).
  • Benefits (Theory):
    • Efficiency: It is computationally "cheaper" than physics-based bullets because it calculates a straight line rather than a moving object with gravity.
    • Precision: It provides instant hit detection, which is ideal for "hitscan" weapons (like lasers or sniper rifles).

II. The Canvas: The User Interface (UI) Layer

In OOAD, the Canvas serves as the Message Layer for the player. It translates the internal state of the game (where the player is looking) into visual feedback (a crosshair).

  • Role of the Canvas: An overlay that sits "on top" of the game world. It hosts UI elements like health bars, crosshairs, and menus.
  • Crosshair Logic: By placing an image (like a red dot) in the exact centre of the Canvas, you provide the player with a visual reference for where the Raycast will be fired.

III. OOAD Integration: The Shooting System

To understand this for an exam, look at how the Weapon and the Target interact as separate classes.

1. The Weapon (The "Aggressor")

The Weapon class handles Behaviour (checking for input) and Function (firing the raycast).

  • Attributes: damage, range, and a reference to the Camera.
  • Logic: When the user clicks, the weapon "asks" the scene: "What did my ray hit?" If the hit object has a Target component, the weapon calls a method on that object: target.TakeDamage(damage).

2. The Target (The "Receiver")

The Target class handles Data (health) and Reaction (dying).

  • Attributes: health.
  • Logic: It waits to be told it took damage. It manages its own state—once health hits zero, it triggers its own destruction (Die()).

IV. Theory Summary: Hit Detection Logic

If you are asked to describe the Targeting and Shooting System in an exam, use these steps:

  1. Input Detection: The system listens for a "Fire" command from the player.
  2. Ray Generation: An invisible ray is projected forward from the centre of the camera view.
  3. Collision Check: The physics engine checks for any objects intersecting that ray within a specific range.
  4. Message Passing: If a target is hit, the weapon script sends a "message" (calls a function) to the target's script.
  5. State Change: The target updates its internal health data. If the data meets a condition (Health ≤ 0), the object is removed from memory.

VI. Analytical Exam Assessment Suite

Review these highly targeted short answer questions and coding evaluation workflows to verify your understanding of Week 4 system parameters.

Question 1: Structural Patterns & Architecture 4 Marks

Explain the explicit differences between Inheritance and the Facade Pattern regarding their architectural relationships, focusing on how a Facade handles subsystem orchestration without code derivation.

Detailed Sample Solution:

  • Relationship Framework: Inheritance models an explicit "Is-A" relationship (e.g., a Student is a Person), focusing on code derivation and extending parental attributes. In contrast, the Facade Pattern leverages composition to model a "Has-A" relationship (e.g., a SchoolFacade has a StudentRecord subsystem), focusing strictly on interface simplification and structural encapsulation.
  • Orchestration Strategy: A Facade class does not inherit attributes or methods from underlying subsystem blocks. Instead, it aggregates instances of discrete subsystem modules internally as composition variables. When an external client issues a single message request to the Facade, it acts as a central coordinator, executing sequentially scoped methods across multiple autonomous subsystems while entirely hiding internal architectural intricacies from the client.
Question 2: Object-Oriented Analysis Systems 3 Marks

Analyse how separating code into Data, Functional, and Behavioural models prevents the formation of "spaghetti code" and supports maintainable game architectures.

Detailed Sample Solution:

Dividing a system layout into distinct data, functional, and behavioural modules isolates responsibilities cleanly:

  1. Data Models isolate state tracking variables entirely from computational mechanics, ensuring an object's internal record (like health or coordinates) cannot be compromised by hidden side effects.
  2. Functional Models manage top-level macro systems, rules, and global processing routines (such as scoring loops and input interpretation profiles).
  3. Behavioural Models dictate localized active reactions and decision algorithms (such as an AI routine chasing targets or physical boundary responses).

This distinct separation ensures scripts remain decoupled; a change to an AI patrolling loop won't compromise structural core database records or global game tracking states.

Question 3: Practical Coding Assessment Task Code Practise

Scenario: Design a micro game targeting engine in Python leveraging strict encapsulation contracts. Create a Target class holding an encapsulated health attribute, a public take_damage() method, and a secondary verification validation method to process elimination out of active system memory. Build a secondary Weapon class that utilizes a message passing reference vector to call operations directly on a target object instance.

Reference Solution Layout:

class Target:
    def __init__(self, initial_health=100):
        # Protected structural encapsulation attribute
        self._health = initial_health
        self._is_alive = True

    def take_damage(self, amount):
        # Safe encapsulated method protecting state variable validation rules
        if not self._is_alive:
            print("Target already neutralized. Action dropped.")
            return
            
        self._health -= amount
        print(f"Target received {amount} points of damage. Current Health: {self._health}")
        self._verify_vitality_bounds()

    def _verify_vitality_bounds(self):
        # Internal validation operation handling state modification
        if self._health <= 0:
            self._health = 0
            self._is_alive = False
            self._trigger_destruction_sequence()

    def _trigger_destruction_sequence(self):
        print("Target health exhausted. Removing object instance from scene memory allocation.")

class Weapon:
    def __init__(self, weapon_damage=35):
        self.damage = weapon_damage

    def fire_at_entity(self, target_instance):
        # Executing message passing via class parameter references
        print("Weapon script triggered. Projecting targeting ray vector...")
        target_instance.take_damage(self.damage)

# Execution Simulation
if __name__ == "__main__":
    combat_target = Target(initial_health=80)
    plasma_rifle = Weapon(weapon_damage=40)

    # Simulating interactive combat routines
    plasma_rifle.fire_at_entity(combat_target)
    plasma_rifle.fire_at_entity(combat_target)
