I. Mapping ERDs to Class Diagrams

Slide 21281

Understanding how to translate an Entity-Relationship Diagram (ERD) into a Class Diagram is a crucial skill in software engineering. This process bridges the gap between database design and object-oriented programming (OOP).

The Step-by-Step Translation Pathway

Step 1 - Entities to Classes: In an ERD, entities represent tables in a database, each containing specific data related to objects within the system. In OOP, these entities map directly to classes. Each class in a class diagram corresponds to an entity in the ERD. For example, a Student entity in an ERD becomes a Student class in OOP. This class will encapsulate all the attributes and behaviours associated with a student.

Step 2 - Attributes to Class Attributes: Attributes in an ERD, which describe properties or characteristics of an entity, map directly to class attributes (fields or properties) in OOP. For example, attributes of a Student entity such as StudentID, Name, and DateOfBirth become attributes of the Student class. These attributes define the data that each instance of the class will hold.

Step 3 - Relationships to Associations: Relationships between entities in an ERD translate to associations between classes in OOP. These associations represent how objects of one class are connected to objects of another class. For example, if a Student entity is related to a Course entity through an enrollment relationship, this relationship becomes an association between the Student class and the Course class. Associations in class diagrams are depicted by lines connecting the classes, often annotated with multiplicity indicators to show cardinality (e.g., one-to-many).

Object-Relational Mapping (ORM)

Object-Relational Mapping (ORM) is a technique that facilitates the conversion between database structures (represented by ERDs) and object-oriented models (represented by class diagrams). ORMs automate the process of mapping database tables to classes, rows to objects, and columns to attributes. This allows developers to interact with the database using the more intuitive and flexible paradigm of OOP.

An ORM framework, such as Django's ORM, manages the translation between the database and the application code. When you define a class in your application, the ORM automatically handles creating the corresponding table in the database. It also manages data retrieval and manipulation, allowing developers to work with database records as objects in their code, simplifying CRUD (Create, Read, Update, Delete) operations.

Primary Keys as Object Identifiers

In an ERD, primary keys are unique identifiers for each entity instance, ensuring that each record can be uniquely distinguished from others. When translating this to OOP, primary keys become unique identifiers for objects. For example, the StudentID attribute in a Student entity serves as the primary key in the database. In the corresponding Student class, this attribute uniquely identifies each Student object, allowing for precise object manipulation and retrieval.

Python Implementation
class Student:
    def __init__(self, student_id, name, date_of_birth):
        self.student_id = student_id  # Primary key mapping
        self.name = name
        self.date_of_birth = date_of_birth

Foreign Keys and Associations

Foreign keys in an ERD establish relationships between different entities by referencing the primary key of another entity. In OOP, these foreign keys translate to references or pointers in classes, linking objects of one class to objects of another. For example, if a Course entity has a foreign key TeacherID that links to a Teacher entity, this relationship translates to a reference in the Course class mirroring the foreign key relationship in the database.

Python Implementation
class Teacher:
    def __init__(self, teacher_id, name):
        self.teacher_id = teacher_id  # Primary key
        self.name = name

Worked Example: Library & Book Linkage

Let's consider a simple example of a library system where Library and Book are two classes that are associated with each other. Each Library has a collection of Books, and each Book belongs to a Library.

  • Classes: Library, Book
  • Attributes: Library: name, books; Book: title, author, library
  • Methods: Library: add_book; Book: set_library
  • Associations: Library has a collection of Book (one-to-many relationship)
library.py
class Book:
    def __init__(self, title, author):
        self.title = title
        self.author = author
        self.library = None  # Holds the explicit Library instance

    def set_library(self, library):
        self.library = library

class Library:
    def __init__(self, name):
        self.name = name
        self.books = [] # Representing a one-to-many relationship

    def add_book(self, book):
        self.books.append(book)
        book.set_library(self)

II. Core Class Relationships

UML uses specific lines and symbols to define how classes interact. Understanding the "strength" of these bonds is critical for design validation.

UML Connectors Architectural Master Summary

Relationship UML Line Type Arrow / Symbol Style Conceptual Meaning & Lifecycle Strength
Association Solid Line None / Open Arrow General structural connection; parts are independent.
Inheritance Solid Line Hollow Triangle Arrowhead "Is-a" relationship; structural subclassing mechanism.
Aggregation Solid Line Hollow (Unfilled) Diamond "Has-a" part-whole link; parts exist independently.
Composition Solid Line Filled (Black) Diamond "Has-a" part-whole link; parts are strictly bound to lifecycle.
Dependency Dashed Line Open Arrowhead "Uses-a" transient constraint; temporary runtime access.

1. Association (General Link)

A basic structural relationship where classes are aware of each other but function completely independently outside of memory pointers.

Association Python Implementation
class Product:
    def __init__(self, name):
        self.name = name

class Customer:
    def __init__(self, name):
        self.name = name
        self.browsed_products = [] # Association reference pointer

2. Aggregation (Part-Whole: "Weak")

Slide 21285

Aggregation is a special form of association that represents a "whole-part" relationship where the part can exist independently of the whole. In this example, we will use a Team and Player classes to demonstrate aggregation. A Team consists of multiple Players, but a Player can exist independently of a Team.

Aggregation Python Implementation
class Player:
    def __init__(self, name):
        self.name = name

class Team:
    def __init__(self, name):
        self.name = name
        self.players = [] # The parts exist independently outside this list container

    def add_player(self, player):
        self.players.append(player)

3. Composition (Part-Whole: "Strong")

Slide 21286

Composition is a stronger form of aggregation where the lifecycle of the part is strictly dependent on the lifecycle of the whole. In this example, we will use a House and Room classes to demonstrate composition. A House consists of multiple Rooms, and a Room cannot exist without a House.

In the context of a composition relationship, it is generally assumed that the "part" cannot exist without the "whole." In this specific example, a Room cannot exist independently without a House. However, a House can conceptually exist without any Rooms, especially when first created before any rooms are added. This setup allows for a house to be created and rooms to be added later. If you want to strictly adhere to the concept that a house must always have rooms, you could enforce that a House must have at least one room upon creation. This design would reflect a stronger dependency between House and Room.

Composition Python Implementation
class Room:
    def __init__(self, name):
        self.name = name

class House:
    def __init__(self, address):
        self.address = address
        self.rooms = [] # Instantiated or fully controlled inside the whole block

    def add_room(self, name):
        # The whole creates the part; part lifecycle tied to host object allocation
        self.rooms.append(Room(name))

4. Inheritance (Generalisation)

Slide 21287

Inheritance is a fundamental concept in object-oriented programming where one class (the subclass) inherits attributes and methods from another class (the superclass), representing an "is-a" relationship. In this example, we will use a Person class as the superclass and Employee and Customer classes as subclasses.

Inheritance Python Implementation
class Person:
    def __init__(self, name, age):
        self.name = name
        self.age = age

class Employee(Person):
    def __init__(self, name, age, employee_id):
        super().__init__(name, age) # explicit parent class init sequence hook
        self.employee_id = employee_id

5. Dependency (Usage Link)

Slide 21288

Dependency in UML class diagrams shows that one class depends on another to function correctly. In this example, let's consider a Report class that depends on a DataFetcher class to fetch data. In UML class diagrams, a dependency relationship is represented by a dashed line with an open arrowhead pointing from the dependent class (the client) to the class it depends on (the supplier).

Dependency Python Implementation
class DataFetcher:
    def fetch_data(self):
        return "Raw System Stream Segment"

class Report:
    def generate(self, fetcher_instance):
        # Parameter validation dependency tracking hook. No persistent attribute reference.
        data = fetcher_instance.fetch_data()
        return f"Report Output: {data}"

III. Professional Standards & Best Practices

  • Encapsulation: Attributes should be marked private (-) in diagrams unless there is a specific reason to make them public (+). Access should be handled via explicit public methods to keep memory configurations well protected.
  • Separation of Concerns: In frameworks like Django (MVT), ensure the View (logic), Model (data), and Template (UI) are kept completely distinct.
  • Multiplicity Placement: Always place the number or symbol at the extreme end of the connecting line closest to the class it describes (e.g., the * goes next to the Book class if a Library has many books).

IV. Library Management System ERD-to-Code Mapping Activity

Slide 21283

Task Constraint: Translate the Library Management System ERD into executable Object-Oriented code structures following the strict cardinality mappings specified below.

Core Architectural Relationships Mapping Schema

  • Member to Loans (1..*): One Member can have multiple Loans. The Member class maintains an active sequence array logging Loan structural objects.
  • Books to Loans (1..*): One Book can appear in multiple Loans across its processing lifetime.
  • Loan to Book and Member (Many-to-One): Each Loan execution instance involves exactly one Book, one Member, and one Librarian helper object.
  • Librarians to Loans (1..*): One Librarian manages multiple Loans over time.

Fully Realised System ERD-to-Code Implementation

library_management_system.py
class Member:
    def __init__(self, member_id, name):
        self.member_id = member_id
        self.name = name
        self.loans = []

    def attach_loan(self, loan):
        self.loans.append(loan)

class Book:
    def __init__(self, isbn, title):
        self.isbn = isbn
        self.title = title
        self.loans = []

    def log_loan(self, loan):
        self.loans.append(loan)

class Librarian:
    def __init__(self, staff_id, name, address, phone):
        self.staff_id = staff_id
        self.name = name
        self.address = address
        self.phone = phone
        self.loans = []  # List to store Loan objects

    def add_loan(self, loan):
        self.loans.append(loan)
        loan.librarian = self

class Loan:
    def __init__(self, loan_date, return_date, book, member, librarian):
        self.loan_date = loan_date
        self.return_date = return_date
        self.book = book
        self.member = member
        self.librarian = librarian  # Reference to a Librarian object
        
        # Enforce reciprocal mapping instantly upon object instantiation
        self.book.log_loan(self)
        self.member.attach_loan(self)

SAQ Question 1 (3 Marks)

Question: In software engineering architectural designs, clarify the operational difference between an Aggregation relationship and a Composition relationship when drawn on a structural UML Class Diagram. Your answer must explicitly address graphical notation markers and class constructor lifecycle dependencies.

Criterion-Referenced Sample Solution:

1. Graphical Notation: Aggregation is represented on a UML layout using a solid line paired with an open (hollow) diamond attached to the "whole" parent class module container. Conversely, Composition requires a solid line accompanied by a filled (solid black) diamond attached to the "whole" parent block class marker.

2. Lifecycle Dependency: Aggregation maps a weak structural linkage where part component entities exist completely independently of the master framework whole lifecycle (e.g., if a Team class is deleted, individual Player object nodes persist safely in system memory). Composition mandates a strong existential bond where sub-part objects cannot survive outside the host parent object boundary; their creation is typical inside the parent container's constructor, ensuring automated cascading destruction when the parent object is swept from system registration limits.

SAQ Question 2 (4 Marks)

Question: Detail how Object-Relational Mapping (ORM) paradigms parse structural relational fields from an Entity-Relationship Diagram (ERD) into an Object-Oriented environment. Explicitly map how Primary Keys, Foreign Keys, and Tables are reassigned inside application runtimes.

Criterion-Referenced Sample Solution:

Object-Relational Mapping abstracts transactional data patterns using a direct schema translation pattern:

  • Tables to Classes: Each distinct record-storing Entity mapped inside a conventional structural ERD converts directly into a standalone operational Class block within the application environment.
  • Primary Keys to Object Identifiers: A unique Primary Key identifier column tracking row instances inside a relational database converts directly into a unique class instance state variable property (e.g., self.student_id) ensuring precise parsing bounds within memory.
  • Foreign Keys to Reference Pointers: Relational reference columns mapping structural constraints via Foreign Keys map cleanly into active class variable references holding memory pointers to actual live instance targets of the associated class object type, substituting plain textual keys with rich object manipulation paths.

class Product:
