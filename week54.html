I. Class Diagram Design Principles

A professional class diagram isn't just a sketch; it must adhere to three core software engineering principles:

1. Encapsulation

Encapsulation is a cornerstone of object-oriented programming (OOP) that focuses on hiding the internal state of an object and restricting access to it through well-defined methods. When drawing class diagrams, encapsulation is represented by using visibility indicators (public, private, protected) for attributes and methods. This ensures that the internal data of a class is protected from direct access by other classes, preventing unintended interference and preserving the integrity of the data.

In a class diagram, encapsulation is depicted by marking attributes as private (using the minus sign '-') and methods that control access to these attributes as public (using the plus sign '+'). This systematically prevents internal corruption and forces modular communication across clear boundaries.

2. Modularity

Modularity refers to dividing a software system into smaller, manageable pieces or modules, each responsible for a specific part of the system's functionality. Well-designed class diagrams promote modularity by clearly defining the responsibilities and boundaries of each class. This separation of concerns makes it easier to understand, develop, test, and maintain each module independently.

In a class diagram, modularity is achieved by identifying distinct classes for different parts of the system and ensuring that each class has a single, well-defined responsibility (The Single Responsibility Principle).

3. Reusability

Reusability is the practice of designing classes and modules that can be used in different parts of a system or in different projects with minimal modification. Well-structured class diagrams promote reusability by defining generic, flexible classes that can be easily adapted and extended.

In a class diagram, reusability is achieved by designing classes with clear interfaces and minimal dependencies on specific implementations.

II. Standard UML Notations

In Unified Modeling Language (UML), class diagrams are used to visually represent the structure of a system by showing its classes, their attributes, methods, and the relationships between them. Understanding the standard notation for these elements is crucial for designing and communicating software architectures.

A UML class diagram consists of classes, each represented as a rectangular box divided into three horizontal compartments:

  1. Top Compartment: Class Name. Contains the class name, typically centred and in bold. It uniquely identifies the structural blueprint type.
  2. Middle Compartment: Attributes. Lists the fields or variables of the class. Format: visibility name: type (e.g., - balance: float). Attributes define the properties or data held by the class instances.
  3. Bottom Compartment: Methods. Lists the methods (also known as operations or functions) of the class. Format: visibility name(parameters): return_type (e.g., + deposit(amount: float): void). Methods define the behaviours or operational capabilities of the class.

Visibility indicators in UML class diagrams show the access level of class members (attributes and methods). They indicate who can access these members, ensuring proper encapsulation and information hiding.

III. Abstract Classes vs. Interfaces

These are "blueprint" tools used to enforce structural uniformity and strict polymorphism rules across multiple distinct subclasses.

1. Abstract Classes (The "Partial Blueprint")

In object-oriented programming (OOP), an abstract class serves as a blueprint for other classes. Abstract classes are designed to be inherited by subclasses that provide implementations for the abstract methods defined in the abstract class.

  • Key Rule: One of the key characteristics of abstract classes is that they cannot be instantiated on their own. This means you cannot create an object directly from an abstract class. Instead, you must create objects from its subclasses that implement the abstract methods.
  • Purpose: To provide a common base with shared attributes and methods that can be utilised by multiple subclasses, ensuring a consistent interface and promoting code reuse. Abstract classes allow you to define methods that must be implemented by all subclasses, enforcing a certain structure and behaviour across different subclasses.
  • Notation: The Class Name is italicised or annotated with <<abstract>>.
  • UML Link: Represented by a solid line with a hollow arrowhead pointing up to the parent (Inheritance relationship).
2. Interfaces (The "Contractual Obligation")

In OOP, an interface defines a contract that implementing classes must follow.

  • Key Rule: Unlike abstract classes, interfaces do not contain any concrete methods or state (attributes). They are purely abstract, serving as a blueprint for what a class must do but not how it does it. They contain zero structural implementation code or internal data fields.
  • Purpose: Interfaces are particularly useful in ensuring that different classes adhere to a specific structure, allowing for consistent behaviour across different classes. This promotes polymorphism, where different classes can be used interchangeably if they implement the same interface.
  • UML Link: Represented by a dashed line with a hollow arrowhead pointing up to the interface block (Implementation / Realization relationship).

IV. Modelling Polymorphism

Polymorphism allows different classes to be treated as instances of a common parent but respond with their own unique behaviours.

1. Method Overriding

In object-oriented programming (OOP), method overriding allows a derived class to provide a specific implementation of a method that is already defined in its base class. This is a powerful feature that enables polymorphism, allowing subclasses to tailor inherited methods to fit their own needs.

When a method in a subclass has the same name, return type, and parameters as a method in its base class, the method in the subclass overrides the one in the base class. This means that when the overridden method is called on an object of the subclass, the implementation in the subclass is executed instead of the one in the base class.

Method overriding is commonly used when a subclass needs to extend or modify the behaviour of a method inherited from a base class. This helps in achieving more specific and refined functionality in subclasses while maintaining a common interface defined by the base class.

2. Dynamic Binding (Late Binding)

Dynamic binding, also known as late binding, is a concept where the method to be invoked is determined at runtime rather than at compile-time. This is essential for achieving polymorphism in OOP. With dynamic binding, the code that calls the method does not need to know the exact type of the object it is dealing with; it just calls the method, and the appropriate method for the actual object type is executed.

Dynamic binding allows objects to be treated as instances of their base class rather than their actual class, enabling flexibility and extensibility in code. This is especially useful in scenarios where the exact type of the object may not be known until runtime, but the operations to be performed are defined by the base class.

Python Code Architectures

1. Abstract Class Blueprint Implementation

To implement an abstract base class pattern in Python, developers leverage the native abc (Abstract Base Classes) library module explicitly.

from abc import ABC, abstractmethod

class Animal(ABC): # Abstract Class definition
    def __init__(self, name):
        self.name = name # Concrete protected instance data state

    @abstractmethod
    def make_sound(self): # Abstract Method (no execution code)
        pass

class Dog(Animal): # Concrete class inheritance path
    def make_sound(self):
        return "Bark"

class Cat(Animal): # Concrete class inheritance path
    def make_sound(self):
        return "Meow"

# Instantiating active memory entities from subclasses
dog = Dog("Buddy")
cat = Cat("Whiskers")
print(f"{dog.name} says {dog.make_sound()}")
print(f"{cat.name} says {cat.make_sound()}")
2. Interface Contract Implementation

Interfaces are handled in Python by defining an abstract class containing exclusively abstract method protocols, housing absolutely no concrete fields, data state definitions, or constructors.

from abc import ABC, abstractmethod

class AnimalInterface(ABC): # Pure Interface Configuration Contract
    @abstractmethod
    def make_sound(self):
        pass

class Dog(AnimalInterface): # Implementation of the structural interface
    def make_sound(self):
        return "Bark"

class Cat(AnimalInterface):
    def make_sound(self):
        return "Meow"

dog = Dog()
cat = Cat()
print(dog.make_sound())
print(cat.make_sound())

Theoretical Short Answer Questions (SAQs)

Question 1 (3 Marks)

Explain the precise design difference between an Abstract Class and an Interface inside a UML Class Diagram framework. Refer explicitly to the structural notations and the line variations used to depict them.

Sample Solution Response:

An abstract class represents a partial software blueprint that can store common data attributes and concrete code definitions alongside abstract methods. In a UML class diagram, it is designated by italics or the <<abstract>> stereotype, and linked via a solid line with a hollow arrowhead to signify inheritance. Conversely, an interface represents a pure behavioral contract containing zero internal data fields or functional code statements. In a UML diagram, an interface is linked to its implementing classes using a dashed line with a hollow arrowhead, representing realization.

Question 2 (4 Marks)

Define the mechanism of Dynamic Binding (Late Binding) within Object-Oriented systems and outline how it works in conjunction with Method Overriding to support true Polymorphism during code execution.

Sample Solution Response:

Dynamic binding is an execution framework where the selection of which exact method to invoke is deferred to runtime rather than decided at compilation. When subclasses employ method overriding, they specify a unique version of a method that exists within their base class. When a polymorphic system executes a call against an abstract type, dynamic binding inspects the absolute concrete object instance residing in system memory at that exact moment. It routes the call to the overridden version matching that specific instance, enabling diverse behaviors via an identical interface structure.

from abc import ABC, abstractmethod

class Vehicle(ABC):
    def __init__(self, registration: str, base_speed: float):
        self.__registration = registration   # Map of private visibility indicator '-'
        self._base_speed = base_speed         # Map of protected visibility indicator '#'

    @abstractmethod
    def calculate_velocity(self) -> float:
        pass

class CombatJet(Vehicle):
    def __init__(self, registration: str, base_speed: float, afterburner_multiplier: float):
        # Invocation via the super() constructor connection path
        super().__init__(registration, base_speed)
        self.__afterburner_multiplier = afterburner_multiplier # Private state attribute

    def calculate_velocity(self) -> float:
        # Explicit execution of method overriding rules
        return self._base_speed * self.__afterburner_multiplier

# Verification Execution Script Routine
if __name__ == "__main__":
    jet_instance = CombatJet("F22-RAPTOR", 450.0, 2.5)
