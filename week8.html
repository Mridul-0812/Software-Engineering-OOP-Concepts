Week 8, Lesson 1: Black Box Software Testing & Automated Testing Frameworks

Comprehensive Theory Study Guide & Relational Analysis — Part 1: Verification Principles & Core Manual Implementations

1. Fundamentals of Software Testing

Software testing is the systematic process of validating and verifying that a software application matches its functional requirements, runs free of bugs, and handles edge-case errors reliably. It is designed to evaluate both correctness (providing the expected output) and robustness (handling invalid states smoothly).

The software testing process isn't limited to finding faults but also focuses on identifying efficiency, usability, and accuracy measures to upgrade the software.

Manual Test Case Architecture

A Test Case is a formal, repeatable script written by engineers to verify a specific software requirement. It represents a specific scenario that testers use to verify that the software behaves as expected. Every professional test case must contain these eight standardised parameters:

Component Purpose
Test Case ID A unique string identifier for traceability (e.g., TC001, TC_ACC_04).
Description A clear, brief summary of the exact functional goal being verified.
Preconditions Any conditions or state the system must be in before execution begins (e.g., user is logged in, account balance is non-zero).
Test Steps The detailed sequence of physical steps, operations, or inputs executed by the tester.
Test Data The specific variables, parameters, or values used during the test execution.
Expected Result The precise, anticipated system output or final state required to consider the test a "Pass".
Actual Result The real system output or state changes observed at runtime (filled out post-execution).
Status The evaluation mark indicating whether the test passed or failed based on comparison: Pass (Expected = Actual) or Fail (Expected ≠ Actual).

2. Black Box Testing Principles

Black Box Testing (also called behavioural or functional testing) treats software as an opaque box. Testers evaluate system functionality entirely from an end-user perspective, without visibility into internal code structures, logic, or algorithms. Testers apply inputs and observe outputs without considering how the software processes those inputs to produce the outcomes. 

+------------------+       +-------------------------+       +-------------------+
|      INPUT       | ----> |     BLACK BOX           | ----> |  EXPECTED OUTPUT  |
|  (Method Args /  |       | (Internal Code Hidden / |       |  (Returned State  |
|   User Actions)  |       |  Encapsulated Boundary) |       |   VS Actual State)|
+------------------+       +-------------------------+       +-------------------+
  • The Input/Output Contract: Inputs are applied to an interface, and the output is compared against the business requirements. If the output matches expectations, the test passes—even if the underlying code is structurally inefficient.
  • OOP Application: In Object-Oriented Programming, Black Box testing validates the public interface of a class (its public methods) while strictly respecting Encapsulation. It checks if invoking a method alters an object's state correctly without manually inspecting its private variables, mangled fields, or internal helper functions.

Core OOP Concept Connection: Encapsulation

Black box testing is the definitive testing approach that maps directly to OOP Encapsulation. Because encapsulation protects data security by sealing properties behind public method gates, black box tests treat the class boundary as impenetrable, proving that the system functions correctly using only exposed operational contracts.

Structured Workflow for Black Box Verification

Workflow Step Architectural Action Required
1. Identify Scenarios Analyze explicit business specifications and requirements to identify functional paths.
2. Define Test Cases Draft formal structural criteria containing the 8 key test components defined above.
3. Execute Test Cases Inject specific test data arrays into the exposed class interfaces and log execution trends.
4. Compare Results Cross-examine actual return objects against your expected result design layouts.
5. Report Issues Isolate failure states and log specific system exceptions to the engineering core.

3. Applied Case Study: System Architecture

To illustrate these testing concepts in practice, we model a basic banking application containing an orchestration class (Bank) and a structural component parts class (Account).

Class Diagram & Structural Dependency Note

The type of connector between the Bank class and the Account class is a structural Association. Specifically, this association forms a Composition relationship, indicating a strict lifetime dependency between the systems.

Composition is a "whole-part" relationship where the parts cannot exist independently outside the parent lifecycle framework. If the Bank instance is destroyed, all nested Account instances contained within it are instantly purged from memory.

5. Exam-Ready Revision & Analysis (Short Answer Questions)

Question 1: Differentiate between software correctness and robustness, and explain how a manual test ledger addresses both criteria when verifying a class interface. [4 Marks]

Sample Solution Response:

Correctness refers to the ability of a software system to deliver expected outputs when fed with valid, conforming operational data matching the system prerequisites (e.g., executing a valid deposit successfully). Conversely, robustness measures the system's capacity to maintain architectural stability and prevent state degradation when exposed to invalid, out-of-bounds inputs or hostile actions (e.g., dropping a negative amount into a withdrawal method).

A manual test ledger maps out both attributes explicitly: testing paths like TC001 track correctness by verifying expected data updates system variables accurately, while edge-case test sequences like TC003 or TC005 directly evaluate robustness by verifying that the class handles illegal operations cleanly and returns failure codes instead of freezing or writing corrupt data to state memory.

Question 2: Analyze why an engineering team would prioritize Black Box methodology over testing internal logic configurations when verifying third-party service dependencies, making explicit reference to OOP Encapsulation. [3 Marks]

Sample Solution Response:

Black Box validation evaluates execution purely from an external viewpoint through the exposed public methods interface of a module. Teams choose this approach because it directly aligns with the rules of Encapsulation, which state that internal properties, class fields, and structural helpers must remain hidden away from outside components.

By focusing strictly on an input-output contract, tests stay independent of changes in underlying data logic or algorithms. This ensures that as long as the module outputs match specifications, the test suite passes, avoiding fragile test breaks if internal code layout changes during updates.

6. Applied OOP Test Script Practice

Here is the implementation of our testing scenario. This framework monitors verification checkpoints using structural OOP workflows, outputting confirmation metrics to the workspace.

Python (manual_test_runner.py)
# =====================================================================
# Target Source Code Modules
# =====================================================================

class Account:
    def __init__(self, account_number, account_holder):
        self.account_number = account_number
        self.account_holder = account_holder
        self.balance = 0.0
        
    def deposit(self, amount):
        if amount > 0:
            self.balance += amount
            return True
        return False
        
    def withdraw(self, amount):
        if 0 < amount <= self.balance:
            self.balance -= amount
            return True
        return False
        
    def get_balance(self):
        return self.balance


class Bank:
    def __init__(self):
        # Composition relationship container
        self.accounts = {}
        
    def create_account(self, account_number, account_holder):
        if account_number not in self.accounts:
            self.accounts[account_number] = Account(account_number, account_holder)
            return True
        return False
        
    def get_account(self, account_number):
        return self.accounts.get(account_number, None)


# =====================================================================
# Programmatic OOP Test Verification Engine
# =====================================================================

class SystemTestOrchestrator:
    def __init__(self):
        self.passed_count = 0
        self.failed_count = 0
        self.error_logs = []

    def assert_equals(self, test_id, actual, expected):
        if actual == expected:
            self.passed_count += 1
            print(f" Check: [PASSED] -> {test_id}")
        else:
            self.failed_count += 1
            err_msg = f"Assertion failure at {test_id}: Received {actual}, Expected {expected}"
            self.error_logs.append(err_msg)
            print(f" ERROR:  [FAILED] -> {test_id}")

    def execute_suite(self):
        print("=" * 60)
        print("INITIALIZING STRUCTURAL BLACK BOX TEST SUITE LOOP")
        print("=" * 60)

        # -------------------------------------------------------------
        # TC001: Valid Deposit Test Sequence
        # -------------------------------------------------------------
        acc_tc001 = Account("771", "TestUser1")
        res_tc001 = acc_tc001.deposit(50.0)
        self.assert_equals("TC001_Deposit_Status", res_tc001, True)
        self.assert_equals("TC001_Deposit_Balance", acc_tc001.get_balance(), 50.0)

        # -------------------------------------------------------------
        # TC002: Valid Withdrawal Test Sequence
        # -------------------------------------------------------------
        acc_tc002 = Account("772", "TestUser2")
        acc_tc002.deposit(100.0)  # Precondition step
        res_tc002 = acc_tc002.withdraw(40.0)
        self.assert_equals("TC002_Withdraw_Status", res_tc002, True)
        self.assert_equals("TC002_Withdraw_Balance", acc_tc002.get_balance(), 60.0)

        # -------------------------------------------------------------
        # TC003: Overdraft Rejection Security Check
        # -------------------------------------------------------------
        acc_tc003 = Account("773", "TestUser3")
        acc_tc003.deposit(100.0)  # Precondition step
        res_tc003 = acc_tc003.withdraw(250.0)
        self.assert_equals("TC003_Overdraft_Status", res_tc003, False)
        self.assert_equals("TC003_Overdraft_Balance", acc_tc003.get_balance(), 100.0)

        # -------------------------------------------------------------
        # TC004: Bank Composition Clashing Prevention Check
        # -------------------------------------------------------------
        bank_test = Bank()
        bank_test.create_account("12345", "Alice") # Baseline state
        res_tc004 = bank_test.create_account("12345", "Bob")
        self.assert_equals("TC004_Duplicate_Acc_Status", res_tc004, False)

        # -------------------------------------------------------------
        # Compilation Reporting Layer
        # -------------------------------------------------------------
        print("=" * 60)
        print("VERIFICATION RUN COMPLETE")
        print(f"Total Successful Checkpoints: {self.passed_count}")
        print(f"Total Discovered Error Anomalies: {self.failed_count}")
        print("=" * 60)

        if self.failed_count == 0:
            print(">>> SUCCESS METRIC: Four distinct tests passed successfully! <<<")
        else:
            print(">>> RUN FAILURE: Error logs isolated below: <<<")
            for error in self.error_logs:
                print(f"[-] {error}")
        print("=" * 60)


if __name__ == "__main__":
    runner = SystemTestOrchestrator()
    runner.execute_suite()
