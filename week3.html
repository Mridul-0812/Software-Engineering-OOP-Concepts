Week 3: Generalisation Paths

Explore the design-level paradigm of structural abstraction, class commonality extraction, and balanced schema organization.

1. Defining Generalisation

Generalisation is the conceptual process of identifying shared characteristics (attributes and behaviours) among different classes and abstracting them into a single, common Superclass.

Expanded Architectural Paradigm: What happens behind the scenes?

In software engineering design, systems grow complex when engineers construct multiple disjoint arrays of files representing standalone domain concepts. Generalisation forces an inspection of these entities before any programmatic syntax is executed. By capturing overlapping features, the developer minimizes duplication at the design phase. It acts as an optimization logic for organizing system definitions.

The Conceptual Workflow:

1
Observe:

Identify multiple classes with overlapping features (e.g., Student and Teacher).

2
Abstract:

Extract the commonalities into a broader category (e.g., Person).

3
Organise:

Establish a hierarchy from General (Superclass) to Specific (Subclass).

Core Concept Breakdown: Generalisation

General Example: Imagine designing an urban transit simulation. You have separate architectural records for ElectricBus, DieselTruck, and HybridSedan. Instead of declaring engine specifications, capacity limits, and speed metrics repeatedly across three separate objects, you observe their commonalities and formulate a unified conceptual framework called Vehicle.

Specific Implementation Example in Python:

generalisation_concept.py Python 
class Vehicle:  # The unified Superclass extracted via Generalisation
    def __init__(self, registration_plate, max_speed):
        self._registration_plate = registration_plate
        self._max_speed = max_speed
        
    def display_base_info(self):
        print(f"Vehicle: {self._registration_plate} | Max Speed: {self._max_speed}km/h")

2. Generalisation vs. Inheritance

While often used interchangeably, they represent two different stages of the software development lifecycle:

Feature Generalisation (Design Phase) Inheritance (Implementation Phase)
Focus Identifying similarities and abstracting them. The technical "is-a" relationship between classes.
Purpose To create a logical, non-redundant schema. To reuse, extend, or modify code via the super() link.
Perspective Bottom-Up: Looking at specific classes to find a common parent. Top-Down: Taking a parent and creating specialised children.

Supplementary Technical Insight

Think of Generalisation as the architect's analytical activity over a drafting board, sketching shared classifications. Inheritance is the structural implementation where Python compilers allocate the memory pointers linking child classes to parent attribute definitions using runtime initialization steps.

3. The Three Levels of Generalisation

Getting the level of abstraction "just right" is a common exam topic. Missteps lead to severe maintenance liabilities.

I. Balanced Generalisation The Goal

Common features are in the superclass; unique features are in the subclasses.

Benefits: High reusability, easy maintenance, and clear scalability.

II. Under-Generalisation Duplication

Occurs when you fail to create a superclass even though commonalities exist.

Issue: Redundancy. If you want to change how "Name" is stored, you must edit the Student, Teacher, and Parent classes individually.

Risk: High maintenance costs and inconsistent data handling.

III. Over-Generalisation Forced Similarity

Occurs when the superclass contains attributes that aren't actually shared by all subclasses.

Example: Putting a role_specific_detail attribute in the Person class to hold a Student ID or a Teacher's Faculty.

Issue: Lack of clarity. It forces subclasses to use generic fields that don't make sense for their specific identity.

Structural Prerequisite: Encapsulation in Hierarchies

General Example: To support proper generalisation limits, classes use Encapsulation by shielding local properties with specialized access visibility (e.g., using a single underscore _ prefix in Python to denote protected attributes). This signals to other parts of the workspace that only the base class and its inherited extensions should read or modify these data scopes.

Specific Python Code demonstrating Encapsulated State Initialization:

encapsulation_boundaries.py Python 
class AccountBlueprint:
    def __init__(self, owner_name, base_routing_id):
        self._owner_name = owner_name                     # Protected structural variable
        self.__base_routing_id = base_routing_id         # Private structure (mangled state)

    def get_owner(self):                             # Accessor exposing managed details safely
        return self._owner_name

4. Practical Implementation: School System

This Python example demonstrates a balanced generalisation where a Person superclass handles the common logic of age and identity.

school_management_system.py Python 
import datetime

# --- SUPERCLASS: The Generalised View ---
class Person:
    def __init__(self, first_name, surname, dob, email):
        self._first_name = first_name
        self._surname = surname
        self._dob = dob # Expects datetime.date object
        self._email = email

    def get_age(self):
        current_year = datetime.datetime.now().year
        return current_year - self._dob.year

    def get_details(self):
        print(f"Name: {self._first_name} {self._surname} | Email: {self._email}")

# --- SUBCLASSES: The Specific Views ---
class Teacher(Person):
    def __init__(self, first_name, surname, dob, email, staff_id, faculty):
        # Initialise the generalised part via explicit super constructor link
        super().__init__(first_name, surname, dob, email)
        # Add specific specialised attributes distinct to this subclass entity
        self._staff_id = staff_id
        self._faculty = faculty

    def get_id(self):
        print(f"Staff ID: {self._staff_id} (Faculty: {self._faculty})")

class Student(Person):
    def __init__(self, first_name, surname, dob, email, student_id):
        super().__init__(first_name, surname, dob, email)
        self._student_id = student_id

    def get_id(self):
        print(f"Student ID: {self._student_id}")

# --- USAGE EXECUTION TARGET ---
t_dob = datetime.date(1985, 10, 10)
teacher = Teacher("John", "Smith", t_dob, "js@school.com", "T99", "Science")
teacher.get_details() # Uses generalised parent infrastructure code execution path
teacher.get_id()      # Uses specialised distinct child method implementation

Key Benefits of Proper Generalisation:

  • Reduced Redundancy: Shared code exists in exactly one place (the superclass).
  • Improved Maintainability: Bug fixes in the superclass automatically apply to all subclasses.
  • Scalability: Adding a new role (e.g., Librarian) is easy because the "Person" foundation is already built.

Term Assessment Preparation: Short Answer Questions (SAQs)
Question 1: Structural Flaw Diagnosis 3 Marks
A junior software design engineer creates an enterprise system with separate classes for ExecutiveManager, ProductionLineWorker, and ContractConsultant. Each class contains independent definitions for '_employee_id', '_tax_file_number', and '_payment_ledger_record'. Explain the structural paradigm failure occurring in this scenario and state how proper generalisation addresses the core issue.
Detailed Sample Solution:

The system suffers from under-generalisation, which causes explicit data structure redundancy across the code workspace. By failing to isolate matching attributes at an earlier engineering phase, any subsequent modification to the core identity format (such as altering the validation schema for _tax_file_number) requires separate modifications in all three separate components. This increases long-term software maintenance overhead and opens vectors for internal synchronization bugs.

To implement proper generalisation, the developer should abstract these matching attributes into a single base superclass named Employee. The specific child classes then inherit these shared variables through inheritance paths, eliminating duplicate definitions completely.

Question 2: Architectural Balancing Mechanics 4 Marks
Differentiate between the structural defects of Over-Generalisation and Under-Generalisation inside an Object-Oriented paradigm. Refer to class cohesion and systemic maintenance boundaries within your response.
Detailed Sample Solution:

Under-Generalisation occurs when the developer fails to distill commonalities among separate entities into a common parent class. This leads to duplicate fields, low maintainability, and severe redundancy, forcing developers to alter identical pieces of logic in multiple files whenever a baseline structural specification changes.

Conversely, Over-Generalisation happens when attributes that are not universal are forced into a unified superclass. For instance, placing a highly specific variable like _flight_clearance_license into a generic Employee class breaks class cohesion. It introduces an architectural abstraction gap, forcing unrelated subclasses (like office administrators) to inherit irrelevant structures that don't match their real-world behaviors.

Practical Code Sandbox Challenge

Review the problem definition below based exclusively on the structural concepts covered in the Week 3 curriculum guidelines.

Scenario: You are tasked with overhauling an inventory system tracker. The system contains a Book class and a VideoGame class. Both track item titles, stock counts, and prices independently, resulting in code redundancy. Refactor this schema into a clean, balanced architecture using a single parent superclass called InventoryItem.

Refactored Solution Target Architecture Python 
# --- PARENT SUPERCLASS VIA DESIGN GENERALISATION ---
class InventoryItem:
    def __init__(self, title, stock_count, base_price):
        self._title = title
        self._stock_count = stock_count
        self._base_price = base_price

    def get_inventory_status(self):
        return f"Item: {self._title} | Stock: {self._stock_count} units | Price: ${self._base_price}"

# --- SPECIALISED SUBCLASSES EXTENDING THE SYSTEM ---
class Book(InventoryItem):
    def __init__(self, title, stock_count, base_price, author_name, page_count):
        super().__init__(title, stock_count, base_price)
        self._author_name = author_name
        self._page_count = page_count

class VideoGame(InventoryItem):
    def __init__(self, title, stock_count, base_price, platform_type):
        super().__init__(title, stock_count, base_price)
        self._platform_type = platform_type
