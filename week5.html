Week 5: Database Design & Entity-Relationship Diagrams (ERDs)

Comprehensive Masterclass Study Node — Modelling Logical Architectures, Normalisation, Key Identifiers, and Object-Oriented Implementations.

1. Fundamentals of ERDs

An Entity-Relationship Diagram (ERD) is a structural diagram used to model the logical architecture of a database. It ensures data integrity and prevents redundancy (repeating data unnecessarily). By formalising data structures graphically before writing code or configuring databases, engineers prevent structural errors that degrade operational execution speeds.

In modern systems engineering, an ERD acts as the ultimate blueprint for data persistence layer interactions. It ensures the system is logically sound, free from data anomalies (errors caused by poor organisation), and normalised. Without mapping structural properties clearly, complex databases risk breaking integrity restraints during deletion, modification, or creation processes.

System Analogy: Think of an ERD as an architectural floor plan for a building. Before pouring physical concrete or routing copper wire, an architect maps exactly where rooms reside and how corridors connect them. If you build without this structural blueprint, you risk placing doors that open into blank brick walls. In software, an ERD stops data elements from being left stranded or duplicated needlessly.

Core Components Mapping

ERDs consist of three main components: Entities (analogous to tables in a database, representing objects or things within the system), Attributes (the properties or characteristics of these entities, similar to columns in a database table), and Relationships (which depict how entities are connected to each other, defining the associations between tables). These components form the absolute foundation of database structure.

Component Software Engineering Definition OOP Analogy Deep-Dive Operational Definition
Entity A definable thing or concept (e.g., Student, Book). Class Significant objects or items within the system. These represent programmatic data blocks that eventually resolve into physical tables within a storage engine.
Attribute A specific property of an entity (e.g., Name, Price). Variable / Field The exact data types, descriptions, and characteristics that an entity holds. These govern what metrics are collected and tracked for individual records.
Relationship How entities interact or connect with one another. Association The explicit logical linkages binding individual entities together, dictating structural execution paths when reading or querying matching blocks.

2. Key Identification (Data Integrity)

To ensure every record in a database is unique and properly connected, we use Keys. Maintaining rigorous Data Integrity means preventing conflicting records, orphaned entries, or data corruption. Keys function as programmatic lookup markers to map memory structures cleanly.

Key Type Technical Definition & Purpose Concrete Systems Example
Primary Key (PK) A unique identifier for an entity instance. Every entity must have one. No two records can ever share the same PK value. Imposes structural isolation. StudentID maps to a unique human. Even if two students share the name "John Smith", their PK isolates their grades and profiles.
Composite Key A primary key made of two or more attributes combined together to guarantee uniqueness. Used when a single property fails to achieve clean uniqueness. In a CourseEnrolment entity, combining StudentID + CourseID creates a unique composite pair, preventing double-enrolments.
Foreign Key (FK) An attribute in one table that serves as a Primary Key in another. It acts as a "link", "pointer", or relational "glue" to connect entities together. Placing TeacherID inside a Course table to map exactly which instructor is currently assigned to deliver that specific syllabus block.

3. Relationship Types & Cardinality

Cardinality defines the numerical constraints of a relationship (how many instances of Entity A can relate to instances of Entity B). It provides numerical indicators along connection paths to constrain database mutations. Coupled with this is Participation, which defines structural dependency rules.

Structural Subtypes of Cardinality

One-to-One (1..1): Each instance of Entity A relates to exactly one instance of Entity B. This is represented by a single line with a "1" or specific perpendicular marks at both ends. Used to enforce secure data isolation.
Example: One Person has exactly one Passport, and that passport belongs to that individual alone.

One-to-Many (1..*): One instance of Entity A relates to multiple instances of Entity B. This forms the backbone of hierarchical structures in enterprise software architectures.
Example: One Teacher manages many Students. A single instructor controls an entire list of downstream learners.

Many-to-Many (*..*): Multiple instances of Entity A relate to multiple instances of Entity B.
Example: Students and Courses. A student takes multiple subjects, while a single course contains an array of students.

Software Engineering Rule: Direct representation of Many-to-Many relationships is highly inefficient and cannot be implemented directly within physical relational tables. They introduce critical querying overheads and structural fragmentation. To fix this, software engineers break them down using a Junction Table (also known as an Associative Entity), which converts the complex layout into two clean, manageable One-to-Many relationships.

Participation & Constraints

Mandatory Participation: An entity must exist in the relationship. If an instance requires an active match to remain valid, it is drawn using a solid line. For example, a Course must have an assigned instructor to exist.

Optional Participation: An entity can exist completely independent of the relationship. It is drawn using a dashed line. For instance, a Teacher can exist within HR systems without currently being assigned to any active teaching course modules.

5. Mapping ERDs to Class Diagrams & Python Logic

Once a database structure is cleanly normalised to prevent structural issues, it acts as a layout map for object-oriented production code. When converting database structures to object classes, remember three core transformation rules:

1. Entities become Classes: Each table translates into an independent Python class.
2. Attributes become Private Variables: Property names map directly to encapsulated internal state fields (prefixed with underscores to protect their structure).
3. Relationships become Object References: One-to-Many pathways translate to objects storing references or full Python lists of other sub-objects (Aggregation or Composition).

Production Scenario: Library Management System

Let us inspect how a real-world scenario maps directly to a high-fidelity database architecture, followed by pure Python code to see it in action.

Business Rules: A Member can register multiple active Loan records. Each individual Loan references exactly one unique Book. A Librarian systematically manages and processes those loans.

Programmatic Blueprint Realisation

Below is the full, clean Python code mapping this explicit database design into memory space using core OOP rules. It uses encapsulation to protect attribute values.

library_system.py 
class Book:
    def __init__(self, book_id: str, title: str, isbn: str):
        self._book_id = book_id  # Protected matching database Primary Key
        self._title = title
        self._isbn = isbn

    def get_title(self) -> str:
        return self._title


class Member:
    def __init__(self, member_id: str, name: str):
        self._member_id = member_id
        self._name = name
        self._active_loans = []  # Captures the 1..* ERD relationship flow via aggregation

    def link_loan(self, loan_object):
        self._active_loans.append(loan_object)


class Loan:
    def __init__(self, loan_id: str, book_instance: Book, member_instance: Member):
        self._loan_id = loan_id
        self._book = book_instance      # Direct object pointer tracking foreign link
        self._member = member_instance  # Injects object instance dependency cleanly
        
        # Synchronise the relationship bi-directionally
        self._member.link_loan(self)

6. Rigorous Examination Assessment Suite

Review these exam-style short answer questions (SAQs). They are mapped directly to assessment rubrics to help you practice writing clear, professional technical explanations.

4 Marks

Short Answer Question 1: Resolving Relational Fragmentation

Explain why a direct Many-to-Many (*..*) relationship cannot be directly implemented within a relational database schema, and describe the systematic mechanism a software engineer must use to resolve this limitation.

Sample Solution Guide (Full Marks Criteria)

Direct Many-to-Many relationships cannot be implemented because they break relational integrity constraints, leading to severe data redundancy and update anomalies. If implemented directly, searching for records would require parsing non-atomic arrays within single cells, slowing query times.

To fix this, an engineer uses a Junction Table (or Associative Entity). This table sits between the two original entities, breaking the Many-to-Many relationship into two clean One-to-Many (1..*) relationships. The Junction Table cross-references data by storing the Primary Keys of both connecting entities as Foreign Keys, ensuring clean data lookup and tracking.

3 Marks

Short Answer Question 2: Architectural Transition Analysis

A system design document shows a One-to-Many (1..*) relationship between a 'Company' entity and an 'Employee' entity. Describe how this relationship is structured when mapping the database schema into an object-oriented Python structure.

Sample Solution Guide (Full Marks Criteria)

When mapping a One-to-Many ERD relationship to Python, the entities are converted into independent, dedicated classes: a Company class and an Employee class.

To capture the "Many" side of the relationship, the Company class uses Aggregation, storing a list attribute (e.g., self._employees = []) to collect multiple employee references. On the "One" side, each instantiated employee object can optionally store a direct pointer reference back to its parent company instance, keeping the reference pathway clear.

ecommerce_model.py 
class Product:
    def __init__(self, product_id: str, sku_code: str, unit_cost: float):
        self._product_id = product_id
        self._sku_code = sku_code
        self._unit_cost = unit_cost

    def get_cost(self) -> float:
        return self._unit_cost


class Customer:
    def __init__(self, customer_id: str, name: str, email: str):
        self._customer_id = customer_id
        self._name = name
        self._email = email
        self._order_history = []  # Maps Customer (1) to Order_Item (*) ERD link

    def add_to_history(self, order_item):
        self._order_history.append(order_item)


class OrderItem:
    def __init__(self, order_item_id: str, customer: Customer, product: Product, quantity: int):
        self._order_item_id = order_item_id
        self._customer = customer      # Resolves customer foreign key association
        self._product = product        # Resolves product foreign key association
        self._quantity = quantity
        
        # Calculate total dynamically using attribute encapsulation paths
        self._total_price = self._quantity * self._product.get_cost()
        
        # Automatically register the order item within the customer's history list
        self._customer.add_to_history(self)
